## Architecture Overview

```
┌──────────────────────────────────────────────────────────────┐
│                    Private Plugin Repos                       │
│  (dec18studios/Grain-By-Greg-DCTL, PhotoChemist-OFX, etc.)  │
│                                                              │
│  Each contains:                                              │
│    manager-release-config.json   ← asset rules, metadata     │
│    GitHub Releases               ← versioned .zip assets     │
└────────────────────────┬─────────────────────────────────────┘
                         │ fetched by CI
                         ▼
┌──────────────────────────────────────────────────────────────┐
│          Dec18-Plugin-Manager (public repo)                   │
│                                                              │
│  .github/workflows/deploy-plugin-manager-pages.yml           │
│    → runs tools/generate-plugin-channel-manifests.mjs        │
│    → generates docs/plugins/<id>/stable.json                 │
│    → updates  docs/plugins/index.json                        │
│    → rewrites download URLs through proxy                    │
│    → deploys to GitHub Pages                                 │
│                                                              │
│  docs/plugins/index.json         ← master catalog            │
│  docs/plugins/<id>/stable.json   ← per-plugin manifest       │
│  src-tauri/src/catalog.rs        ← embedded fallbacks         │
│  src/main.js                     ← frontend UI/filtering      │
└────────────────────────┬─────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────┐
│          GitHub Pages (dec18studios.github.io)                │
│   plugins/index.json                                         │
│   plugins/<id>/stable.json                                   │
│   updates/latest.json   (Tauri self-updater feed)            │
└────────────────────────┬─────────────────────────────────────┘
                         │ fetched at runtime
                         ▼
┌──────────────────────────────────────────────────────────────┐
│              Tauri Desktop App                                │
│  1. Fetches index.json → gets list of plugins                │
│  2. Fetches each stable.json → gets versions & download URLs │
│  3. Compares with local install stamps                       │
│  4. Renders dashboard with tabs/filters                      │
│  5. Downloads via proxy (license-gated)                      │
│  6. Installs to correct path per platform                    │
└──────────────────────────────────────────────────────────────┘
```

### Download Flow

```
App → POST download request with license token
   → Cloudflare Worker (download proxy) validates Ed25519 signature
   → Proxy fetches asset from private GitHub repo (using GITHUB_PAT)
   → Streams .zip back to app
   → App extracts and installs to installPath
```

## Quick Reference — What to Touch

### Files to modify (in Dec18-Plugin-Manager repo)

| # | File | Action | Required? |
|---|------|--------|-----------|
| 1 | **Private plugin repo**: `manager-release-config.json` | Create config in the plugin's own repo | **Yes** |
| 2 | **Private plugin repo**: GitHub Release | Create a tagged release with properly-named .zip asset | **Yes** |
| 3 | `docs/plugins/index.json` | Add entry with `pluginId`, `displayName`, `type`, `licenseTier`, `manifestUrl`, `category` | **Yes** |
| 4 | `.github/workflows/deploy-plugin-manager-pages.yml` | Add `"Repo-Name\|plugin-id"` to the `PLUGINS=()` array | **Yes** |
| 5 | `docs/plugins/<plugin-id>/stable.json` | Auto-generated by CI, but need a seed for embedded fallback | Recommended |
| 6 | `src-tauri/src/catalog.rs` | Add `EMBEDDED_*` constant + match arm for offline fallback | Recommended |

### Identity Rules

| Field | Convention | Examples |
|-------|-----------|----------|
| `pluginId` | lowercase-kebab-case, must match across ALL files | `grain-by-greg-dctl`, `photochemist`, `ibkeymaster-ofx` |
| `type` | Exact casing matters in index.json | `DCTL`, `OFX`, `App`, `3rd-party` |
| `licenseTier` | Two values | `free`, `subscription` |
| `bundleIdentifier` | Reverse-domain | `com.dec18studios.GrainByGreg` |
| `assetPattern` | Regex matching the GitHub release asset filename | `Grain.By.Greg.*\.zip$` |

## Adding a DCTL Plugin

### Step 1: Private Plugin Repo — `manager-release-config.json`

Create this file at the root of the plugin's GitHub repo (e.g., `dec18studios/My-New-DCTL`):

```json
{
  "pluginId": "my-new-dctl",
  "displayName": "My New DCTL",
  "releaseRepo": "dec18studios/My-New-DCTL",
  "minManagerVersion": "0.1.0",
  "hostProcesses": ["Resolve", "DaVinci Resolve"],
  "requiredFamilies": ["universal"],
  "category": "Color Grading",
  "description": "Short description of what this DCTL does.",
  "licenseTier": "subscription",
  "tags": ["color", "grading", "DCTL"],
  "assetRules": [
    {
      "family": "universal",
      "platform": "macos",
      "arch": "universal",
      "assetPattern": "My.New.DCTL.*\\.zip$",
      "packageType": "zip",
      "bundleName": "My New DCTL.dctl",
      "bundleIdentifier": "com.dec18studios.my-new-dctl",
      "installPath": "/Library/Application Support/Blackmagic Design/DaVinci Resolve/LUT/DCTL",
      "installMode": "file-browse"
    },
    {
      "family": "universal",
      "platform": "windows",
      "arch": "x86_64",
      "assetPattern": "My.New.DCTL.*\\.zip$",
      "packageType": "zip",
      "bundleName": "My New DCTL.dctl",
      "bundleIdentifier": "com.dec18studios.my-new-dctl",
      "installPath": "C:\\ProgramData\\Blackmagic Design\\DaVinci Resolve\\Support\\LUT\\DCTL",
      "installMode": "file-browse"
    },
    {
      "family": "universal",
      "platform": "linux",
      "arch": "x86_64",
      "assetPattern": "My.New.DCTL.*\\.zip$",
      "packageType": "zip",
      "bundleName": "My New DCTL.dctl",
      "bundleIdentifier": "com.dec18studios.my-new-dctl",
      "installPath": "/opt/resolve/LUT/DCTL",
      "installMode": "file-browse"
    }
  ]
}
```

**DCTL-specific notes:**
- `installMode` is always `"file-browse"` — the user picks the install folder on first install
- `requiredFamilies` is `["universal"]` — DCTLs are cross-platform (single `.dctl`/`.dctle` file)
- All three platform rules use the same `assetPattern` because the same .zip works everywhere
- `bundleName` must match the actual filename inside the .zip

### Step 2: Create a GitHub Release

1. Tag the release (e.g., `v1.0.0`)
2. Upload a `.zip` asset whose name matches `assetPattern`
   - Example: `My-New-DCTL-v1.0.0.zip` matches `My.New.DCTL.*\.zip$`
3. Ensure the release is **not a draft** and **not a prerelease** (for stable channel)
4. Optionally add release highlights in the body using HTML comments:
   ```
   <!-- manager-highlights:start -->
   - Added new color mode
   - Fixed gamma curve
   <!-- manager-highlights:end -->
   ```

### Step 3: Add to `docs/plugins/index.json`

Add an entry to the `plugins` array (keep alphabetical by `displayName`):

```json
{
  "pluginId": "my-new-dctl",
  "displayName": "My New DCTL",
  "type": "DCTL",
  "licenseTier": "subscription",
  "manifestUrl": "https://dec18studios.github.io/Dec18-Plugin-Manager/plugins/my-new-dctl/stable.json",
  "category": "Color Grading"
}
```

### Step 4: Add to Deploy Workflow

In `.github/workflows/deploy-plugin-manager-pages.yml`, add to the `PLUGINS=()` array:

```bash
"My-New-DCTL|my-new-dctl"
```

Format: `"GitHub-Repo-Name|pluginId"` — the repo name is used to fetch from `dec18studios/<Repo-Name>`.

### Step 5: Seed the Local Manifest (for embedded fallback)

Create `docs/plugins/my-new-dctl/stable.json` with a minimal manifest. CI will overwrite this on first deploy, but having a seed ensures the embedded fallback works:

```json
{
  "pluginId": "my-new-dctl",
  "displayName": "My New DCTL",
  "description": "Short description.",
  "category": "Color Grading",
  "type": "DCTL",
  "licenseTier": "subscription",
  "tags": ["color", "grading", "DCTL"],
  "version": "1.0.0",
  "releaseDate": "2026-04-08T00:00:00Z",
  "releaseNotesUrl": "https://github.com/dec18studios/My-New-DCTL/releases/tag/v1.0.0",
  "platforms": [
    {
      "platform": "macos",
      "arch": "universal",
      "downloadUrl": "https://dec18-download-proxy.dec18studios.workers.dev/v1/dec18studios/My-New-DCTL/releases/download/v1.0.0/My-New-DCTL-v1.0.0.zip",
      "sha256": "",
      "packageType": "zip",
      "bundleName": "My New DCTL.dctl",
      "bundleIdentifier": "com.dec18studios.my-new-dctl",
      "installPath": "/Library/Application Support/Blackmagic Design/DaVinci Resolve/LUT/DCTL",
      "minManagerVersion": "0.1.0",
      "hostProcesses": ["Resolve", "DaVinci Resolve"],
      "installMode": "file-browse"
    }
  ],
  "availableVersions": []
}
```

### Step 6: Add Embedded Fallback to `catalog.rs`

In `src-tauri/src/catalog.rs`:

1. Add a constant (near the top, with the other `EMBEDDED_*` constants):
   ```rust
   const EMBEDDED_MY_NEW_DCTL: &str = include_str!("../../docs/plugins/my-new-dctl/stable.json");
   ```

2. Add a match arm in the `embedded_manifest_for()` function:
   ```rust
   "my-new-dctl" => EMBEDDED_MY_NEW_DCTL,
   ```

### Step 7: Commit, Push, Verify

```bash
git add docs/plugins/index.json \
        docs/plugins/my-new-dctl/stable.json \
        .github/workflows/deploy-plugin-manager-pages.yml \
        src-tauri/src/catalog.rs
git commit -m "feat: add My New DCTL to plugin catalog"
git push
```

The push triggers the Pages deploy, which will:
1. Fetch the `manager-release-config.json` from the private repo
2. Fetch the GitHub releases
3. Generate `docs/plugins/my-new-dctl/stable.json` with real SHA256 hashes and proxy URLs
4. Update `docs/plugins/index.json` (preserving `type` and `licenseTier`)
5. Deploy to GitHub Pages

## Adding an OFX Plugin

Same process as DCTL with these differences:

### Config Differences (`manager-release-config.json`)

```json
{
  "pluginId": "my-new-ofx",
  "displayName": "My New OFX",
  "releaseRepo": "dec18studios/My-New-OFX",
  "minManagerVersion": "0.1.0",
  "hostProcesses": [
    "Resolve", "DaVinci Resolve", "Fusion", "Fusion Studio",
    "Nuke", "NukeX", "NukeStudio", "Hiero"
  ],
  "requiredFamilies": ["macos"],
  "category": "Color",
  "description": "Description of the OFX plugin.",
  "tags": ["OFX", "color"],
  "assetRules": [
    {
      "family": "macos",
      "platform": "macos",
      "arch": "universal",
      "assetPattern": "My-New-OFX.*macOS.*universal.*\\.zip$",
      "packageType": "zip",
      "bundleName": "MyNewOFX.ofx.bundle",
      "bundleIdentifier": "com.dec18studios.MyNewOFX",
      "installPath": "/Library/OFX/Plugins"
    }
  ]
}
```

**OFX-specific notes:**
- `installMode` defaults to `"bundle"` (elevated copy to system path) — omit or set explicitly
- `requiredFamilies` — set to `["macos"]` if macOS-only, or `["macos", "windows", "linux"]` for cross-platform
- Each platform needs a separate asset rule with platform-specific `assetPattern`, `installPath`, and `arch`
- `bundleName` must end in `.ofx.bundle`
- `hostProcesses` should include all OFX hosts the plugin supports
- OFX plugins may need separate assets per platform (unlike DCTLs)

### OFX Platform Install Paths

| Platform | Install Path |
|----------|-------------|
| macOS | `/Library/OFX/Plugins` |
| Windows | `C:\Program Files\Common Files\OFX\Plugins` |
| Linux | `/usr/OFX/Plugins` |

### Multi-Platform OFX Asset Rules

If the OFX plugin ships for all platforms:

```json
"requiredFamilies": ["macos", "windows", "linux"],
"assetRules": [
  {
    "family": "macos",
    "platform": "macos",
    "arch": "universal",
    "assetPattern": "MyPlugin.*macOS.*universal.*\\.zip$",
    "installPath": "/Library/OFX/Plugins",
    ...
  },
  {
    "family": "windows",
    "platform": "windows",
    "arch": "x86_64",
    "assetPattern": "MyPlugin.*[Ww]indows.*x86_64.*\\.zip$",
    "installPath": "C:\\Program Files\\Common Files\\OFX\\Plugins",
    ...
  },
  {
    "family": "linux",
    "platform": "linux",
    "arch": "x86_64",
    "assetPattern": "MyPlugin.*linux.*x86_64.*\\.tar\\.gz$",
    "packageType": "tar.gz",
    "installPath": "/usr/OFX/Plugins",
    ...
  }
]
```

### Index Entry

```json
{
  "pluginId": "my-new-ofx",
  "displayName": "My New OFX",
  "type": "OFX",
  "licenseTier": "subscription",
  "manifestUrl": "https://dec18studios.github.io/Dec18-Plugin-Manager/plugins/my-new-ofx/stable.json",
  "category": "Color"
}
```

## Adding a 3rd-Party Plugin

3rd-party plugins follow the same pattern, but the release repo belongs to a different GitHub org/user.

### Key Differences

1. The `releaseRepo` points to the external repo: `"thirdparty-user/Their-Plugin"`
2. The `PLUGIN_REPOS_PAT` secret must have access to that repo (or it must be public)
3. `type` is `"3rd-party"` in `index.json`
4. You still create a `manager-release-config.json` — but it lives in the Dec18 manager repo under `docs/plugins/<plugin-id>/` since you can't commit to the 3rd-party repo

### Alternative: Local Config

If you can't add `manager-release-config.json` to the 3rd-party repo, place it in the manager repo instead. The deploy workflow fetches configs from the private plugin repos, so for 3rd-party repos, modify the workflow to read configs from a local path.

Current workaround — the deploy workflow can be extended with a local-config fallback:

```bash
# In the PLUGINS loop, check for local config if remote fetch fails
if ! curl -fsSL ... -o "$CONFIG_FILE" 2>/dev/null; then
  LOCAL_CONFIG="docs/plugins/${PLUGIN_ID}/manager-release-config.json"
  if [ -f "$LOCAL_CONFIG" ]; then
    cp "$LOCAL_CONFIG" "$CONFIG_FILE"
  else
    echo "::notice::${REPO}: no config found — skipping"
    continue
  fi
fi
```

### Index Entry

```json
{
  "pluginId": "their-plugin-dctl",
  "displayName": "Their Plugin",
  "type": "3rd-party",
  "licenseTier": "free",
  "manifestUrl": "https://dec18studios.github.io/Dec18-Plugin-Manager/plugins/their-plugin-dctl/stable.json",
  "category": "Color Grading"
}
```

## File-by-File Reference

### `manager-release-config.json` (in plugin repo)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `pluginId` | string | **Yes** | Unique kebab-case ID — must match everywhere |
| `displayName` | string | **Yes** | Human-readable name shown in UI |
| `releaseRepo` | string | **Yes** | `owner/repo` for GitHub API calls |
| `minManagerVersion` | string | **Yes** | Minimum manager version required |
| `hostProcesses` | string[] | **Yes** | Apps that must be closed before install (empty = none) |
| `requiredFamilies` | string[] | **Yes** | Asset families that must exist for a release to be valid |
| `category` | string | No | Grouping category (`"Color Grading"`, `"Film Emulation"`, etc.) |
| `description` | string | No | Short description for UI |
| `licenseTier` | string | No | `"free"` or omit (defaults to `"subscription"`) |
| `tags` | string[] | No | Searchable tags |
| `infoUrl` | string | No | External product page URL |
| `assetRules` | object[] | **Yes** | Platform-specific download/install rules (see below) |

### `assetRules[]` Entry

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `family` | string | **Yes** | Family group: `"universal"`, `"macos"`, `"windows"`, `"linux"` |
| `platform` | string | **Yes** | Target OS: `"macos"`, `"windows"`, `"linux"` |
| `arch` | string | **Yes** | CPU architecture: `"universal"`, `"x86_64"`, `"aarch64"` |
| `assetPattern` | string (regex) | **Yes** | Regex to match GitHub release asset filenames |
| `packageType` | string | **Yes** | `"zip"` or `"tar.gz"` |
| `bundleName` | string | **Yes** | Expected filename inside the archive |
| `bundleIdentifier` | string | **Yes** | Reverse-domain ID for install stamps |
| `installPath` | string | **Yes** | Default install directory |
| `installMode` | string | No | `"bundle"` (default, system path) or `"file-browse"` (user picks) |

### `docs/plugins/index.json`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `pluginId` | string | **Yes** | Must match config and manifest |
| `displayName` | string | **Yes** | Shown in tabs |
| `type` | string | **Yes** | `"DCTL"`, `"OFX"`, `"App"`, `"3rd-party"` — controls tab filtering |
| `licenseTier` | string | **Yes** | `"free"` or `"subscription"` — controls license gating |
| `manifestUrl` | string | **Yes** | URL to the `stable.json` on GitHub Pages |
| `category` | string | No | Used for category dropdown filter |

### `src-tauri/src/catalog.rs` — Embedded Constants

Two changes needed per plugin:


## The Deploy Pipeline

### What Triggers a Redeploy

| Trigger | When |
|---------|------|
| Push to `docs/**` | Any change to index.json, local manifests |
| Push to workflow file | Changes to deploy-plugin-manager-pages.yml |
| After successful build | Workflow_run after build-plugin-manager.yml succeeds |
| Manual dispatch | `gh workflow run deploy-plugin-manager-pages.yml` |
| Repository dispatch | External `plugin-release` event |

### What the Deploy Does (step by step)

1. **Checkout** `main` branch
2. **For each plugin** in `PLUGINS=()` array:
   a. Fetch `manager-release-config.json` from the private plugin repo (via `PLUGIN_REPOS_PAT`)
   b. Fetch all GitHub releases from the plugin repo
   c. Filter to stable releases (non-draft, non-prerelease, with assets)
   d. Run `generate-plugin-channel-manifests.mjs`:
      - Match release assets against `assetPattern` regex
      - Compute SHA256 of each matched asset
      - Extract release highlights from HTML comments
      - Write `docs/plugins/<id>/stable.json` (and `beta.json` if applicable)
      - **Update `docs/plugins/index.json`** — merges with existing entry, preserving `type`/`licenseTier`
3. **Rewrite download URLs** — replace `https://github.com/...` with proxy URLs (`https://dec18-download-proxy...`)
4. **Copy** `docs/` → `pages-dist/`
5. **Generate updater feed** — `tools/generate-updater-feed.mjs` creates `pages-dist/updates/latest.json`
6. **Deploy** `pages-dist/` to GitHub Pages

### Critical: The `updateIndex()` Merge Behavior

The manifest generator's `updateIndex()` function **merges** new data with existing index entries using spread:

```javascript
const existing = existingIndex >= 0 ? entries[existingIndex] : {};
const nextEntry = {
  ...existing,           // preserves type, licenseTier, etc.
  pluginId,              // from config
  displayName,           // from config
  manifestUrl,           // computed
  category: category || existing.category || null,
};
```

This means `type` and `licenseTier` set in `index.json` are preserved across deploys — they only need to be set once. However, if a plugin is **new** (no existing entry), these fields must already be in `index.json` before the first deploy.

## Common Pitfalls

### 1. pluginId Mismatch

The `pluginId` must be **identical** across all four locations:
- `manager-release-config.json` → `"pluginId"`
- `docs/plugins/index.json` → `"pluginId"`
- `deploy-plugin-manager-pages.yml` → second part of `"Repo|pluginId"`
- `catalog.rs` → match arm string

A mismatch creates **duplicate entries** in the deployed index.

### 2. assetPattern Doesn't Match Release Asset

The `assetPattern` is a JavaScript regex tested against the **exact filename** of each GitHub release asset. Common mistakes:
- Asset named `My-Plugin-v1.0.0.zip` but pattern is `My\.Plugin.*\.zip$` (hyphens vs dots)
- Forgetting the `$` anchor
- Case sensitivity — the regex is case-sensitive

**Verify:** Check the actual asset name with:
```bash
gh release view v1.0.0 --repo dec18studios/My-Plugin --json assets --jq '.assets[].name'
```

### 3. Missing `type` or `licenseTier` in index.json

If you add a plugin to the workflow but forget to set `type`/`licenseTier` in `index.json`, tab filtering breaks (plugin appears only in "All Tools" tab).

### 4. Forgetting the Embedded Fallback

Without the `EMBEDDED_*` constant in `catalog.rs`, the plugin won't appear when the app is offline or on first launch before the network fetch completes.

### 5. `requiredFamilies` vs Actual Assets

If `requiredFamilies` includes `"windows"` but there's no Windows asset rule or no Windows release asset, the manifest generator will **fail** with:
```
Error: <pluginId>: release v1.0.0 has no matching packaged assets
```

Only include families you actually ship.

### 6. Bundle Permissions (OFX on macOS)

After manual install of `.ofx.bundle`, ensure `chmod -R 755` — Resolve silently fails to load plugins it can't read.

### 7. Stale OFX Plugin Cache

After fixing OFX bundle issues, delete:
```
~/Library/Application Support/Blackmagic Design/DaVinci Resolve/OFXPluginCacheV2.xml
```
Then restart Resolve to force a re-scan.

## Streamlined Workflows

### Workflow 1: One-Command New DCTL

For an agent adding a DCTL that already has a GitHub release:

```bash
# Variables
PLUGIN_ID="my-new-dctl"
DISPLAY_NAME="My New DCTL"
REPO_NAME="My-New-DCTL"
CATEGORY="Color Grading"
TYPE="DCTL"
LICENSE="subscription"  # or "free"
BUNDLE_NAME="My New DCTL.dctl"
ASSET_PATTERN="My.New.DCTL.*\\.zip$"

# 1. Create config in private repo (via GitHub API)
# 2. Add to index.json
# 3. Add to deploy workflow PLUGINS array
# 4. Seed local manifest
# 5. Add embedded fallback to catalog.rs
# 6. Commit and push
# 7. Trigger deploy
gh workflow run deploy-plugin-manager-pages.yml
```

### Workflow 2: Verify a Deploy

After pushing changes, verify the deployed catalog:

```bash
# Check deploy status
gh run list --workflow=deploy-plugin-manager-pages.yml --limit 1

# When complete, verify the live index
curl -s https://dec18studios.github.io/Dec18-Plugin-Manager/plugins/index.json \
```

### Workflow 3: Update Private Repo Config via API

When you can't clone the private repo, use the GitHub API:

```bash
# Read current config
gh api repos/dec18studios/My-Plugin-Repo/contents/manager-release-config.json \
  --jq '.content' | base64 -d

# Get SHA for update
SHA=$(gh api repos/dec18studios/My-Plugin-Repo/contents/manager-release-config.json --jq '.sha')

# Write updated config
echo '<new JSON content>' > /tmp/config.json
CONTENT=$(base64 -i /tmp/config.json)
gh api repos/dec18studios/My-Plugin-Repo/contents/manager-release-config.json \
  -X PUT \
  -f message="update manager config" \
  -f content="$CONTENT" \
  -f sha="$SHA"
```
